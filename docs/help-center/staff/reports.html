What this page is for

Use this guide when you need something more precise than a dashboard widget. It covers the stable reporting flow: choose the right entity, shape the report, save it, and schedule it for regular delivery.

Before you start

  • Know what question the report needs to answer before you open the builder.
  • Choose the right source entity first; changing it later resets much of the setup.
  • Have recipient email addresses ready if you plan to create a schedule.

Steps

  1. Choose the source entity first. In Report Builder, start with the entity that matches the operational question you are answering.
  2. Select fields and filters. Add only the fields you need, then layer filters so the output reflects the exact working set.
  3. Use grouping and aggregations deliberately. These controls are most helpful when you need totals, averages, counts, or other summarized views.
  4. Save the report when it will be reused. Give it a name that tells staff what question it answers, not just what module it came from.
  5. Start from templates when speed matters. Use Report Templates for common report categories, then adapt the template instead of rebuilding from zero.
  6. Use Saved Reports for repeat work. Load a saved report when you need the same logic again, and use sharing or public-link controls only if your process requires them.
  7. Schedule recurring delivery when needed. Use Scheduled Reports to set recipients, frequency, timing, and output format, then return there to run, pause, or review history.
Screenshot: Report Builder with entity buttons and field selection

Changing the entity clears major parts of the current report definition. Set the entity before you invest time in fields and filters.

What happens next

Once a report is saved or scheduled, it becomes reusable operational infrastructure. The next step is usually to incorporate it into a team routine, a meeting, or a recurring delivery list.

Common mistakes

  • Starting with filters before choosing the correct entity.
  • Saving reports with vague names that other staff cannot interpret.
  • Scheduling a report before you confirm recipients and timezone details.
  • Using a dashboard for a question that really needs report filters or aggregation.

Need help?

If a report output looks wrong, capture the entity, filters, grouping choices, and whether the issue appears in the builder, saved report, or schedule view. That makes it much easier to distinguish a setup mistake from a real data issue.
